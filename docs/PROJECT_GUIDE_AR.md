# مشروع FullDelivery - الدليل الفني الكامل

## 1. فكرة المشروع

FullDelivery هو منصة توصيل متكاملة من الميل الأخير (last-mile delivery) تهدف إلى ربط المتاجر والسائقين والعملاء من خلال تطبيق موحد. المنصة تسمح للمتاجر بإنشاء وإدارة طلبات التوصيل، وللسائقين بقبول وتنفيذ التوصيلات مع تتبع GPS في الوقت الفعلي والتحقق عبر OTP/صورة/توقيع، وللعملاء بتتبع طلباتهم في الوقت الفعلي من خلال نظام متكامل للدردشة والإشعارات والمحفظة الإلكترونية.

### الأهداف
- إنشاء نظام توصيل موثوق وفعال في منطقة الشرق الأوسط
- توفير تجربة سلسة لجميع الأطراف (متجر، سائق، عميل)
- أتمتة التسوية المالية بين الأطراف
- توفير شفافية كاملة في تتبع الطلبات

### الرؤية
خلق أكثر أنظمة التوصيل موثوقية وكفاءة في الشرق الأوسط من خلال منصة موحدة تربط جميع الأطراف المعنية بعملية التوصيل مع تتبع شفاف وتواصل فوري وتسوية مالية آلية.

### نموذج العمل
تعمل FullDelivery على نموذج عمولة:
- **المتاجر** تدفع عمولة منصة عن كل توصيل (مخزنة في `platform_commission`)
- **السائقون** يكسبون رسوم توصيل + مكافآت (`driver_earnings` + `reward_bonus`)
- **المحفظة** تتم المعاملات بشكل ذري — يتم إيداع أرباح السائق فور إتمام التوصيل عبر RPC آمن مع قفل الصف (`SELECT FOR UPDATE`)
- **طرق الدفع**: كاش، بطاقة، محفظة

## 2. البنية الحالية (Current Architecture)

### نظرة عامة

```
تطبيق موبايل (React Native + Expo)
        │
        ▼
    Supabase (PostgreSQL, Auth, Realtime, Edge Functions)
        │
        ├── PostgreSQL ─── 22 جدول، 62 ميجريشن، RLS كامل
        ├── Auth ─────────── بريد إلكتروني + كلمة مرور
        ├── Realtime ─────── postgres_changes + presence
        └── Edge Functions ── assign-order, send-notification
```

### المكدس التقني

- **الإطار الرئيسي:** React Native 0.83.6 + Expo SDK 55
- **التوجيه:** Expo Router 4.x (توجيه يعتمد على الملفات)
- **اللغة:** TypeScript 5.9
- **قاعدة البيانات:** Supabase (PostgreSQL)
- **إدارة الحالة:** Zustand 5 (3 مخازن)
- **الخادم:** TanStack React Query 5
- **الخرائط:** react-native-maps + OpenStreetMap (CartoDB)
- **الإشعارات:** Expo Notifications + Supabase Edge Functions
- **الترجمة:** i18next + react-i18next (عربي + إنجليزي)

### هيكل التوجيه

التطبيق يستخدم Expo Router مع توجيه يعتمد على الملفات. ثلاث وحدات أساسية:

```
(app)/
├── (store)/     ─── المتجر (3 تبويبات: الطلبات، الرسائل، الملف الشخصي)
├── (driver)/    ─── السائق (5 تبويبات: الرئيسية، الطلبات، الدردشة، المحفظة، الملف الشخصي)
├── (customer)/  ─── العميل (3 تبويبات: الطلبات، الرسائل، الملف الشخصي)
├── (chat)/      ─── الدردشة المشتركة
└── (notifications)/ ─── الإشعارات المشتركة
```

كل وحدة تحتوي على شاشات مخفية (href: null) للتفاصيل والإعدادات وعمليات إنشاء الطلب.

## 4. شرح قاعدة البيانات

### الجداول الأساسية (22 جدول)

#### `profiles`
الجدول المركزي لهوية المستخدم. مرتبط 1:1 مع `auth.users`. يحتوي على: الدور (role)، الاسم الكامل، رقم الهاتف، الصورة الرمزية.

#### `stores`
معلومات المتجر: الاسم، السجل التجاري، العنوان، الإحداثيات، الشعار، حالة النشاط.

#### `store_staff`
موظفو المتجر الإضافيون مع صلاحيات محددة (إنشاء طلبات، عرض تقارير).

#### `customers`
سجلات العملاء (يمكن أن يكونوا غير مسجلين — `profile_id` قابل للإلغاء). يتم إنشاؤهم عبر `ensure_customer_by_phone` RPC.

#### `customer_addresses`
عناوين التوصيل المحفوظة لكل عميل. تحتوي على: التصنيف (منزل/عمل)، النص الكامل للعنوان، الإحداثيات، الشقة، الطابق، معلم بارز، ملاحظات.

#### `drivers`
ملف السائق: السيارة، لوحة الأرقام، الحالة (متصل/غير متصل/مشغول)، معلومات البنك، التقييم، عدد التوصيلات.

#### `driver_documents`
مستندات السائق للتحقق (رخصة، تسجيل مركبة، هوية). مع حالة التحقق.

#### `shipment_types`
أنواع الشحنات المرجعية: صغير (1 كجم)، متوسط (10 كجم)، كبير (30 كجم)، قابل للكسر (20 كجم).

#### `delivery_orders` - **الجدول المركزي**
أكثر من 40 عمودًا. يحتوي على كل معلومات الطلب: المتجر، العميل، السائق، عنواني الاستلام والتوصيل، التفاصيل، الرسوم، الحالة، التواريخ، OTP.

**حالات الطلب (9 قيم):**
```
pending → published → driver_accepted → driver_arrived_store → picked_up → on_the_way → driver_arrived_destination → delivered
                                                                                                                                 → cancelled (من أي حالة)
```

#### `order_assignments`
محاولات تعيين السائق للطلب. حالة واحدة `accepted` فقط لكل طلب — الباقي `expired` تلقائيًا.

#### `order_status_history`
سجل تدقيق لجميع تغييرات الحالة. يستخدم لعرض الخط الزمني في شاشة تفاصيل الطلب.

#### `delivery_issues`
تقارير المشكلات أثناء التوصيل (عميل غير متوفر، عنوان خاطئ، مشكلة في المتجر، إلخ).

#### `driver_locations`
بيانات GPS الزمنية أثناء التوصيلات النشطة. تحتوي على: driver_id، order_id، خط الطول/العرض، الدقة، وقت التسجيل.

#### `conversations`
محادثة واحدة لكل طلب (1:1 مع delivery_orders).

#### `conversation_participants`
أعضاء كل محادثة (الدور: متجر/سائق/عميل).

#### `messages`
الرسائل مع النوع (نص/صورة/صوت/فيديو/ملف/موقع/نظام) و metadata بنظام JSONB.

#### `notifications`
الإشعارات داخل التطبيق مع نوع الإشعار والعنوان والنص والبيانات (تحتوي على order_id للتنقل).

#### `notification_templates`
قوالب الإشعارات ثنائية اللغة (عربي + إنجليزي) مع متغيرات.

#### `wallets`
محفظة واحدة لكل مستخدم (تنشأ تلقائيًا). الرصيد من نوع DECIMAL(12,2) مع علامة is_frozen.

#### `wallet_transactions`
جميع المعاملات المالية. الرصيد الإيجابي = إيداع، السلبي = سحب.

#### `push_tokens`
رموز الإشعارات الفورية للأجهزة (FCM/APNs).

## 5. نظام التوثيق (Authentication)

### التدفق الكامل:

1. **التسجيل**: المستخدم يسجل عبر `useAuthStore.signUp()` → Supabase Auth تنشئ سجل `auth.users`
2. **إنشاء الملف الشخصي**: يتم إدراج سجل في جدول `profiles` مع الدور المختار
3. **JWT Claim**: trigger يضيف `user_role` إلى JWT
4. **توجيه إلى تسجيل الدخول**: يتم توجيه المستخدم إلى شاشة تسجيل الدخول
5. **تسجيل الدخول**: `useAuthStore.initialize()` تجلب الجلسة والملف الشخصي
6. **التحقق من المسار**: `useAuthGuard` تتحقق من حالة المصادقة واكتمال الملف
7. **إذا كان السجل الخاص بالدور مفقودًا**: يتم التوجيه إلى شاشة الإعداد (setup)
8. **شاشة الإعداد**: تنشئ السجل الخاص بالدور (متجر/سائق/عميل) → `refreshProfile()` → توجيه إلى التطبيق
9. **إنشاء المحفظة**: `trg_create_wallet_on_profile_insert` تنشئ المحفظة تلقائيًا

### قواعد Route Guard:
- المستخدم غير الموثّق → يتم توجيهه إلى `/login`
- المستخدم الموثّق بدون سجل دور → يتم توجيهه إلى `/setup`
- المستخدم الموثّق مع سجل دور كامل → يتم توجيهه إلى وحدته (`/(app)/(store)` أو `/(app)/(driver)` أو `/(app)/(customer)`)

## 7. نظام المتجر (Store System)

### المكتمل:
- **لوحة التحكم**: إحصائيات في الوقت الفعلي، الطلبات الأخيرة، إجراءات سريعة
- **إنشاء الطلب**: 6 خطوات (البحث عن العميل → اختيار العنوان → تفاصيل الطلب → مراجعة → إنشاء)
- **قائمة الطلبات**: بحث، تصفية حسب الحالة، ترقيم الصفحات، تحديث فوري
- **تفاصيل الطلب**: خط زمني، تتبع مباشر على الخريطة، إلغاء/نسخ/تعديل
- **دليل العملاء**: قائمة العملاء من سجل الطلبات
- **الإعدادات**: إعدادات المتجر
- **الملف الشخصي**: عرض وتحرير معلومات المتجر
- **تحرير العنوان**: تحديد موقع المتجر على الخريطة

### المتبقي:
- شاشة المحفظة (الرصيد وسجل المعاملات)
- شاشة التقارير (التحليلات)
- شاشة تفاصيل العميل (`customer/[phone]`)
- تحسينات إدارة الطلبات (تعديل الطلب، إجراءات جماعية)

## 8. نظام السائق (Driver System)

### المكتمل (100%):
- **لوحة التحكم**: إحصائيات، الطلبات المتاحة، الإجراءات السريعة
- **الطلبات المتاحة**: قائمة بالطلبات المنتظرة مع المسافة والسعر
- **طلباتي**: الطلبات النشطة والمكتملة والملغاة
- **قبول الطلب**: تأكيد القبول مع تفاصيل الدفع
- **الوصول إلى المتجر**: تأكيد الوصول
- **تأكيد الاستلام**: تصوير الدليل، ملاحظات
- **في الطريق**: خريطة حية مع مسار وتتبع، ETA
- **تأكيد التوصيل**: OTP/صورة/توقيع
- **ملخص التوصيل**: تفاصيل الأرباح
- **الإبلاغ عن مشكلة**: اختيار نوع المشكلة مع وصف
- **المحفظة**: الرصيد، الفترات (اليوم/الأسبوع/الشهر)، سجل المعاملات
- **المكافآت**: الإنجازات مع أشرطة التقدم
- **المستندات**: رفع وتحميل (رخصة، تسجيل مركبة، هوية)
- **الملف الشخصي**: معلومات السائق الكاملة
- **الإعدادات**: اللغة، المظهر، الخصوصية، المساعدة، معلومات عن التطبيق
- **الحالة**: حالة التوثيق والمستندات

### البنية:
كل انتقالات حالة الطلب تتم عبر RPCs آمنة (SECURITY DEFINER):
```
accept_order → arrive_at_store → confirm_pickup → start_delivery → arrive_at_destination → complete_delivery
```

## 10. نظام المشرف (Admin System)

الوضع الحالي: دعم على مستوى API فقط عبر RLS bypass.

- دالة `is_admin()` تتحقق من JWT claim
- المشرف لديه صلاحية SELECT/INSERT/UPDATE/DELETE على جميع الجداول
- RPCs: `admin_soft_delete_user` و `admin_restore_user`
- **لا توجد واجهة مستخدم للمشرف** — لا توجد شاشات بعد

---

## 11. نظام المحفظة (Wallet System)

### التنفيذ الحالي:
- محفظة واحدة لكل مستخدم (تُنشأ تلقائيًا عند إنشاء الملف الشخصي)
- الرصيد من نوع DECIMAL(12,2) مع is_frozen
- جميع المعاملات عبر RPC `add_wallet_transaction` مع `SELECT FOR UPDATE` (قفل الصف)
- يتم إيداع أرباح السائق تلقائيًا في `complete_delivery` RPC
- **مكتمل للسائق**: عرض الرصيد، سجل المعاملات، فترات (يوم/أسبوع/شهر)
- **مفقود للمتجر**: لا توجد شاشة محفظة للمتجر بعد
- **مفقود**: لا يتم إيداع أرباح المتجر تلقائيًا

### التنفيذ المستقبلي:
- إضافة شاشة محفظة للمتجر
- إضافة RPC لإيداع أرباح المتجر
- نظام السحب (payout) للمتجر والسائق

## 14. الإشعارات (Notifications)

نظام من جزئين:

1. **Database triggers**: `notify_order_status_change` تُدرج سجلات في `notifications` عند تغيير حالة الطلب
2. **Edge Function**: `send-notification` تقرأ `push_tokens` وترسل عبر Expo Push API

### داخل التطبيق:
- `useNotificationsList` hook مع اشتراك Realtime
- شاشة الإشعارات مع علامة "غير مقروء"
- النقر على الإشعار ينقل إلى الشاشة المناسبة:
  - `new_message` → `(chat)/[orderId]`
  - أنواع أخرى → `(role)/[orderId]`

### أنواع الإشعارات (6):
`order_update`, `new_message`, `driver_assignment`, `nearby_order`, `complaint_update`, `system`

## 15. الوقت الفعلي (Realtime)

### النمط الإلزامي:
```typescript
const channel = supabase.channel('name')
  .on('postgres_changes', { event: '*', schema: 'public', table: 'table', filter: '...' }, callback)
  .on('postgres_changes', { event: '*', schema: 'public', table: 'table2' }, callback2)
  .subscribe();
return () => { supabase.removeChannel(channel); };
```

### الجداول المنشورة في `supabase_realtime`:
messages, conversations, conversation_participants, delivery_orders, drivers, notifications, wallet_transactions, order_status_history, driver_locations

**القاعدة:** يجب تسجيل ALL `.on()` قبل `.subscribe()`. لا يمكن إضافة `.on()` بعد الاشتراك.

## 16. الدردشة (Chat)

- محادثة واحدة لكل طلب (1:1 مع `delivery_orders`)
- RPC `ensure_conversation` تنشئ أو تجلب معرف المحادثة
- Hook `useConversation`: رسائل فورية مع مؤشر الكتابة
- Trigger `add_driver_to_conversation`: يضيف السائق تلقائيًا عند تعيينه
- الميزات: تجميع الرسائل حسب الوقت، الطوابع الزمنية، علامات القراءة

- **التنقل:** Google Maps deep links للتوجيه خطوة بخطوة
- **6 استخدامات:** en-route، store/[orderId]، customer/[orderId]، AddressForm، LocationPickerScreen، complete-address

## 18. نظام العناوين (Address System)

### التدفق الكامل:

#### للمتجر (إنشاء طلب):
1. إدخال رقم هاتف العميل → `get_customer_info` RPC
2. إذا كان العميل موجودًا → `get_customer_addresses` RPC → عرض العناوين المحفوظة
3. اختيار عنوان موجود أو إضافة عنوان جديد → `add_customer_address` RPC (مع التحقق من صلاحية المتجر)
4. العودة إلى شاشة الاختيار → التحديث التلقائي → اختيار العنوان الجديد → متابعة
5. إذا كان العميل جديدًا → شاشة اختيار الموقع على الخريطة

#### للعميل (إدارة العناوين):
1. شاشة قائمة العناوين (عرض/تعديل/حذف/تعيين افتراضي)
2. إضافة عنوان → AddressForm مع اختيار الموقع على الخريطة
3. تعديل عنوان → تحميل البيانات الحالية → حفظ
4. شاشة الإعداد الأولى → إذا لم يكن هناك عناوين → توجيه إلى إدخال العنوان

## 19. دورة حياة الطلب (Order Lifecycle)

```
إنشاء الطلب (pending)
       │
       ▼
نشر الطلب (published) ← Edge Function: assign-order
       │
       ▼
يقبل السائق (driver_accepted) ← accept_order RPC
       │
       ▼
يصل السائق إلى المتجر (driver_arrived_store) ← arrive_at_store RPC
       │
       ▼
يؤكد الاستلام (picked_up) ← confirm_pickup RPC
       │
       ▼
في الطريق (on_the_way) ← start_delivery RPC
       │
       ▼
يصل إلى الوجهة (driver_arrived_destination) ← arrive_at_destination RPC
       │
       ▼
توصيل ناجح (delivered) ← complete_delivery RPC
         │
         ├── التحقق: OTP / صورة / توقيع
         ├── إيداع المحفظة تلقائيًا
         └── تحديث إحصائيات السائق
```

الإلغاء مسموح من أي حالة.

## 20. تدفق البيانات (Database Flow)

### عند إنشاء طلب:
```
[المتجر] → CreateOrderStore → Supabase insert delivery_orders
    → trigger sync_assigned_driver
    → trigger notify_new_order (إشعار للسائقين)
    → trigger create_order_notifications
    → Edge Function assign-order (إنشاء order_assignments)
```

### عند إتمام التوصيل:
```
[السائق] → complete_delivery RPC (SECURITY DEFINER)
    → تحديث حالة الطلب إلى delivered
    → add_wallet_transaction (إيداع أرباح السائق)
    → تحديث total_deliveries في drivers
    → trigger notify_order_status
    → trigger create_order_notifications
    → إشعار للمتجر والعميل
```

### تدفق المحفظة:
```
add_wallet_transaction(p_wallet_id, p_amount, p_type, ...)
    → BEGIN
    → SELECT balance FROM wallets WHERE id = p_wallet_id FOR UPDATE
    → التحقق من الرصيد الكافي (للسحب)
    → INSERT INTO wallet_transactions
    → UPDATE wallets SET balance = balance + p_amount
    → COMMIT
```

## 23. القرارات المعمارية الهامة

### لماذا يدفع السائق للمتجر أولاً؟
هذا ليس صحيحًا — النموذج المالي في FullDelivery هو:
- السائق يدفع للمتجر نقدًا (لأن خيار الدفع cash هو الافتراضي)
- العميل يدفع للسائق عند التوصيل
- المنصة تخصم العمولة تلقائيًا عند إتمام التوصيل
- مستقبلاً: خيارات الدفع بالبطاقة والمحفظة ستغير هذا النموذج

### لماذا RPC (وليس direct insert)؟
- **الأمان**: SECURITY DEFINER يسمح بتجاوز RLS مع التحقق من الصلاحية
- **الذرية**: `add_wallet_transaction` تستخدم `SELECT FOR UPDATE` لقفل الصف ومنع السباق
- **المنطق التجاري**: transitions تحتاج منطقًا تجاريًا (تحقق من الحالة السابقة، تحديث الطابع الزمني، إنشاء سجل التدقيق)
- **قاعدة**: أي عملية تحتاج أكثر من خطوة SQL واحدة يجب أن تكون RPC

### لماذا SECURITY DEFINER؟
لأن بعض الاستعلامات تحتاج تجاوز RLS للوصول إلى جداول متعددة. مثال: `accept_order` يحتاج قراءة الطلب، إنشاء assignment، وتحديث الطلب — وكلها تحت صلاحيات مختلفة.

### تصميم المحفظة
- رصيد واحد لكل مستخدم (DECIMAL(12,2))
- is_frozen لتعليق المعاملات
- كل المعاملات عبر add_wallet_transaction مع قفل الصف
- يتم إنشاء المحفظة تلقائيًا عند إنشاء الملف الشخصي
- **قرار مهم**: الرصيد يُحسب ديناميكيًا كمجموع المعاملات (وليس تخزينه فقط) — لضمان الاتساق

### Single Source of Truth
- Supabase هو المصدر الوحيد للحقيقة — لا بيانات وهمية، لا mock
- مخازن Zustand للحالة المؤقتة فقط (الجلسة، تدفق إنشاء الطلب، الإعدادات)
- كل التعديلات المالية → RPCs آمنة

### المكونات المشتركة
- مكونات واجهة المستخدم العامة في `src/components/ui/`
- مكونات الملف الشخصي مشتركة عبر الأدوار الثلاثة
- مكونات خاصة بكل دور في مجلدها الخاص
- لا تكرار للمكونات أبدًا

### Hooks المشتركة
- `use-auth.ts` — منطق المصادقة والتوجيه
- `use-chat.ts` — الدردشة الفورية
- `use-conversations.ts` — قائمة المحادثات
- `use-notifications-list.ts` — الإشعارات
- `use-push-token.ts` — تسجيل الإشعارات الفورية

## 24. قواعد المطور (Developer Rules)

### القواعد الإلزامية:

1. **لا تكرار**: لا تكرر المكونات، الـ hooks، أو منطق الأعمال
2. **لا بيانات وهمية**: لا تستخدم بيانات mock أو قيم وهمية — دائماً استخدم Supabase
3. **TypeScript صارم**: 0 أخطاء TypeScript مطلوبة قبل الـ commit
4. **RPC أولاً**: أي تغيير في حالة الطلب أو معاملة مالية → RPC
5. **الميجريشن أولاً**: دائماً أنشئ SQL migration قبل كتابة TypeScript types
6. **توليد الأنواع**: استخدم `supabase gen types typescript --local` — لا تكتب الأنواع يدويًا
7. **لا console.log**: ممنوع في كود الإنتاج
8. **Realtime pattern**: `.channel()` → ALL `.on()` → `.subscribe()` — لا تضف `.on()` بعد الاشتراك
9. **router.back()**: دائماً مع `canGoBack()` ومسار احتياطي
10. **استخدم نظام التصميم**: دائماً استخدم `useColors()`، `spacing.*`، `fontSize.*` — لا ألوان أو مسافات مكتوبة يدويًا
11. **لا تلمس الوحدات الأخرى**: لا تعدل وحدات السائق/العميل/الدردشة/الإشعارات أثناء العمل على المتجر
12. **التحقق من التنسيق**: تأكد من `npx tsc --noEmit` يمر بـ 0 أخطاء

## 25. دليل الاختبار (Testing Guide)

### قبل رفع أي كود:

1. **TypeScript**: `npx tsc --noEmit` — 0 أخطاء
2. **فحص الاتصال**: تحقق من أن جميع استدعاءات Supabase تستخدم RPCs صحيحة
3. **Realtime**: تأكد من نمط `.channel()` → `.on()` → `.subscribe()` الصحيح
4. **التنقل**: تأكد من أن جميع `router.back()` محمية بـ `canGoBack()`
5. **RLS**: اختبر أن المستخدم العادي لا يمكنه الوصول إلى بيانات ليست له
6. **RPCs**: اختبر أن RPCs تعمل مع مستخدمين مخوّلين وغير مخوّلين

### اختبار دورة حياة الطلب:

1. إنشاء طلب من المتجر
2. التحقق من ظهور الطلب للسائقين
3. قبول الطلب من السائق
4. متابعة جميع الحالات حتى التوصيل
5. التحقق من إيداع المحفظة

### اختبار إنشاء العميل والعنوان:

1. إدخال رقم هاتف موجود → عرض العناوين
2. إضافة عنوان جديد → العودة إلى الاختيار → الاختيار التلقائي
3. إدخال رقم هاتف جديد → اختيار الموقع على الخريطة

## 26. سير عمل Git

### الفروع:
- **main** — كود جاهز للإنتاج، يمر TypeScript بـ 0 أخطاء
- **feature/*** — ميزات جديدة (مثال: feature/store-wallet)
- **fix/*** — إصلاح الأخطاء (مثال: fix/realtime-channel-race)

### تسمية الـ commits:
- `feat: description` — ميزة جديدة
- `fix: description` — إصلاح خطأ
- `refactor: description` — إعادة هيكلة
- `db: description` — ميجريشن قاعدة بيانات أو RPC
- `docs: description` — توثيق
- `chore: description` — صيانة

### Pull Requests:
- PR يجب أن يمر بـ 0 أخطاء TypeScript
- PR يجب أن لا يكسر الميزات الموجودة (regression)
- مراجعة الشيفرة إلزامية
- لا force push إلى main

### قواعد الدمج:
- squash merge إلى main
- حذف الفرع بعد الدمج
- التأكد من أن جميع الميجريشنات قابلة للتطبيق والتّراجع
